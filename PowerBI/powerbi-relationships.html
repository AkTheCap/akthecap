01 — Foundation

What is a Table Relationship?

In Power BI, you almost always have multiple tables. A "relationship" is the link between two tables using a common column (called a key). Think of it like a bridge.

Beginner

IPL Analogy

Imagine two scoresheets:

  • match_info — has one row per match (Match 335989, Match 335991…)
  • toss_winners — has who won the toss in each match

Both have a column called match_number. Power BI uses this to join them — like pinning two scoresheets together by the match number.

02 — Cardinality

Cardinality: How Many Rows Match?

Cardinality describes how many rows in one table relate to how many rows in another. Power BI has 3 types.

Beginner

One-to-One (1:1)

1 : 1

Each row in Table A matches exactly one row in Table B, and vice versa.

match_info
match_number (PK)venue
335989Wankhede
335991Eden Gardens
336008Chepauk
toss_winners
match_number (FK)toss_winner
335989MI ✓
335991KKR ✓
336008CSK ✓
✅ Real use case

Each match has exactly one toss result. match_info ↔ toss_winners is a perfect 1:1. Use this when two tables have the same unique key and you split them for organisation.

Intermediate

One-to-Many (1:*) — Most Common!

1 : *

One row in Table A corresponds to many rows in Table B. This is the backbone of most Power BI models.

✅ Real use case

Match 335989 has hundreds of balls bowled. So match_info (1 row per match) → ball_by_ball (many rows per match) is a classic 1:Many. The "1" side always has unique values.

Expert

Many-to-Many (*:*) — Handle with Care!

* : *

Many rows in Table A match many rows in Table B. This can cause double-counting and unexpected results.

playing_XI (player in match)

⚠️ Why M:M is dangerous

If you join playing_XI to players_details on player directly, Power BI creates a Many-to-Many. Aggregations like SUM of runs can get multiplied incorrectly. Always resolve M:M using a bridge table or a proper dimension table.

✅ Solution: Use a unique Dimension table

Create a dim_players table with one unique row per player. Then: playing_XI (*) → dim_players (1) and ball_by_ball (*) → dim_players (1). No M:M needed.

03 — Cross-Filter Direction

Cross-Filter Direction

When you select/filter a value in one table, which other tables get filtered? That's cross-filter direction. Think of it as "which way does the filter flow?"

Beginner

Single Direction →

Filter flows in one direction only — from the "1" (dimension) side to the "*" (fact) side. This is the default and recommended setting.

match_info
Dimension (1)
→
ball_by_ball
Fact (*)
Example: Select "CSK" in match_info → ball_by_ball shows only CSK balls. But you cannot filter match_info from ball_by_ball.
Intermediate

Both Directions ⇄

Filter flows in both directions. Useful but risky — can cause circular dependency errors and unexpected filtering.

match_info
⇄
ball_by_ball
Example: Select a ball in ball_by_ball → match_info also gets filtered to show only that match. Used in your screenshot (toss_winners ↔ match_info with Both).
Expert

When to Use "Both" — The Rule

✅ Use BOTH when:
  • You have a 1:1 relationship
  • You need bidirectional slicing (e.g. toss analysis vs match result)
  • The tables are small lookup/dimension tables
  • You're building a role-playing dimension
❌ Avoid BOTH when:
  • You have M:M relationships
  • Multiple paths exist between tables (causes circular refs)
  • You're on large fact tables (performance hit)
  • You don't need reverse filtering

05 — The Error You Hit

Circular Reference / Ambiguous Paths

This is the error you mentioned. It happens when Power BI finds more than one path to connect two tables — and doesn't know which one to use.

🔴 What causes it in IPL data?

Look at your model: match_info connects to toss_winners AND both might connect to match_teams. If you also have batting_team and bowling_team both pointing to a teams table with Both-direction filters, you create an ambiguous loop:

// Power BI sees two paths from batting_team to bowling_team: batting_team → match_info ← bowling_team batting_team → teams_info ← bowling_team // Power BI doesn't know which path to use → ERROR

❌ Problem Setup

batting_team ⇄ match_info
bowling_team ⇄ match_info
batting_team ⇄ teams_info ⇄ bowling_team
↑ Circular loop!

💡 Why it breaks

With "Both" direction on multiple paths, the filter doesn't know where to stop. It bounces between tables infinitely.

Power BI deactivates one relationship and shows a warning, or throws an error in DAX.

✅ Fixes

  1. Change "Both" → Single
  2. Use USERELATIONSHIP() in DAX
  3. Mark one relationship as Inactive
  4. Redesign as a Star Schema
✅ The Fix for Your Screenshot

In your toss_winners → match_info relationship, you used Both direction + 1:1. That's fine by itself. The problem occurs when another table also has a "Both" relationship to either of these. Fix: change all fact-table relationships to Single, and only use Both on true 1:1 dimension lookups.

-- When you MUST use two paths, use USERELATIONSHIP in DAX: CALCULATE( SUM(ball_by_ball[runs_scored]), USERELATIONSHIP(match_info[match_number], batting_team[match_number]) )

Star Schema Rules

  1. Fact table in the center (ball_by_ball, batting scores)
  2. Dimension tables on the outside (teams, players, dates)
  3. All relationships are 1:Many (dim → fact)
  4. All filters are Single direction (dim → fact)
  5. No table connects to another dimension directly

07 — Expert Level

Expert Tips & DAX Tricks

Expert

USERELATIONSHIP()

When you have inactive relationships (e.g., a match has both a batting team and a bowling team, both pointing to dim_teams), use this in DAX to specify which path to use:

// Runs scored BY batting team Batting_Runs = CALCULATE( SUM(ball_by_ball[runs_scored]), USERELATIONSHIP( batting_team[match_number], ball_by_ball[match_number] ) )
Expert

CROSSFILTER()

Override filter direction inside a specific measure without changing the model relationship:

// Force bidirectional filter for this measure only Matches_Won = CALCULATE( COUNTROWS(match_info), CROSSFILTER( match_info[winner], teams_info[team_name], Both ) )
Expert

Role-Playing Dimensions

Your model has batting_team and bowling_team — both refer to team names. Instead of two tables, use one dim_teams with two inactive relationships:

-- Active: ball_by_ball[batting_team] → dim_teams[team] -- Inactive: ball_by_ball[bowling_team] → dim_teams[team] -- In DAX, activate the inactive one: Bowling_Team_Wickets = CALCULATE( SUM(ball_by_ball[is_wicket]), USERELATIONSHIP( ball_by_ball[bowling_team], dim_teams[team_name] ) )
Expert

Performance Rules

  1. Integer keys are faster than string keys — use match_number (int) not match_date (string)
  2. Avoid Both on large fact tables — it creates hidden filter propagation
  3. Date Table must always be marked as official Date Table in Power BI
  4. Hide FK columns in fact tables — show only from dimension side
  5. Referential integrity assumption speeds queries — only enable if data is clean
